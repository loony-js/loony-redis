Great — designing Redis structure properly early saves a lot of pain later.

Redis doesn’t enforce schema, so **you must design it intentionally**.

Below are **production-grade Redis schema design best practices** used in real backend systems.

---

# 🔹 1️⃣ Always Use Namespaces (Key Prefixing)

Redis has no tables — your key name _is your structure_.

### ✅ Recommended Format

```
app:module:entity:id[:field]
```

### Example (E-commerce App)

```
ecom:user:1001
ecom:order:5001
ecom:cart:1001
ecom:session:abc123
```

### Why?

- Prevents key collisions
- Easy pattern scanning
- Clear logical grouping
- Safer multi-team development

---

# 🔹 2️⃣ Prefer DB 0 Only (Avoid Logical DBs)

Although Redis supports multiple databases:

```bash
SELECT 1
```

In real production:

- Redis Cluster supports only DB 0
- Monitoring tools assume DB 0
- Easier DevOps management

👉 Use prefixes instead of multiple DBs.

---

# 🔹 3️⃣ Choose the Right Data Structure

This is critical for performance & memory.

## 🔹 Strings (Cache / Simple Values)

Best for:

- Cached objects (JSON)
- Counters
- Tokens

```
SET cache:user:1001 "{json data}" EX 300
```

---

## 🔹 Hashes (Structured Objects)

Better memory efficiency than storing many strings.

```
HSET ecom:user:1001 name "Sankar" email "s@mail.com" age 25
```

Best for:

- User profiles
- Settings
- Metadata

---

## 🔹 Lists (Queues / Order History)

```
LPUSH ecom:queue:emails job1
```

Use for:

- Job queues
- Logs
- Event buffers

---

## 🔹 Sets (Unique Collections)

```
SADD ecom:user:1001:roles "admin"
```

Use for:

- Tags
- Followers
- Unique memberships

---

## 🔹 Sorted Sets (Ranking / Time-based)

```
ZADD ecom:leaderboard 100 user1
```

Use for:

- Leaderboards
- Rate limiting
- Time-based indexing

---

# 🔹 4️⃣ Use TTL Strategically

Not everything should live forever.

### Good Candidates for Expiry:

- Cache data
- Sessions
- OTP codes
- Rate limit counters

Example:

```
SET session:abc123 user1 EX 3600
```

Avoid TTL for:

- Core business data
- Critical financial records

---

# 🔹 5️⃣ Avoid Big Keys

Problem example:

```
HSET users_all ...
```

If one key grows too large:

- Slower operations
- Replication delays
- Memory spikes

Better:

```
user:1001
user:1002
user:1003
```

Rule of thumb:

- Keep values < few MB
- Keep collections < hundreds of thousands per key

---

# 🔹 6️⃣ Avoid KEYS \* in Production

Instead use:

```
SCAN 0 MATCH ecom:user:*
```

Design your schema so you rarely need to scan large datasets.

---

# 🔹 7️⃣ Design for Access Patterns (Most Important)

Don’t design based on data structure.
Design based on how you read/write data.

Example:

If you frequently:

- Get user by ID → `user:1001`
- Get orders by user → maintain index

```
SET ecom:order:5001 {...}
SADD ecom:user:1001:orders 5001
```

Redis does NOT support joins — you must model relationships manually.

---

# 🔹 8️⃣ Handle Relationships Explicitly

Relational DB:

```
SELECT * FROM orders WHERE user_id = 1
```

Redis equivalent:

```
SMEMBERS ecom:user:1:orders
GET ecom:order:5001
```

You maintain secondary indexes yourself.

---

# 🔹 9️⃣ Multi-Tenant Design

For SaaS systems:

```
app:tenant:1:user:1001
app:tenant:2:user:1001
```

Never mix tenant data without namespace isolation.

---

# 🔹 🔟 Version Your Keys (Advanced)

When changing structure:

Old:

```
user:1001
```

New:

```
v2:user:1001
```

Prevents breaking old deployments.

---

# 🔹 1️⃣1️⃣ Memory Awareness

Check usage:

```
MEMORY USAGE key
INFO memory
```

Best practices:

- Use hashes instead of many small keys
- Avoid large JSON blobs when partial updates are needed
- Consider compression for big values

---

# 🔹 1️⃣2️⃣ Common Production Patterns

### Cache Pattern

```
cache:product:5001
```

### Rate Limiting

```
rate:login:ip:12345
```

### Distributed Lock

```
lock:order:5001
```

### Session Store

```
session:user:1001
```

---

# 🚀 Example: Clean E-commerce Redis Design

```
ecom:user:1001                (hash)
ecom:order:5001               (hash)
ecom:user:1001:orders         (set)
ecom:product:9001             (hash)
ecom:cart:1001                (hash)
ecom:leaderboard              (sorted set)
ecom:session:abc123           (string + TTL)
```

---

# 🧠 Golden Rules Summary

1. Use clear prefixes
2. Use DB 0 only
3. Choose correct data structure
4. Model relationships manually
5. Avoid large keys
6. Use TTL intentionally
7. Design for access patterns
8. Never rely on KEYS in production
